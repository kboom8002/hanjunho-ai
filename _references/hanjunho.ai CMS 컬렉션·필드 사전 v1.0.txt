hanjunho.ai CMS 컬렉션·필드 사전 v1.0
부제: 웹플로우 개편을 위한 CMS 구조, 필드 정의, 운영 규칙 최종안
1. 문서 목적
이 문서는 hanjunho.ai의 Webflow CMS를 혼동 없이 설계·구현·운영하기 위한 기준 문서다.목적은 세 가지다.
첫째, 디자이너와 개발자가 어떤 컬렉션이 왜 필요한지를 같은 기준으로 이해하게 한다.둘째, 콘텐츠 운영자가 카드, 브리핑, 프로필, 팩트체크를 추가할 때 필드 입력 규칙을 일관되게 적용하게 한다.셋째, 홈, 클리어맵, 상세페이지, 인증센터, 팩트체크, 브리핑 페이지에서 어떤 필드가 어디에 노출되는지를 명확히 고정한다.
이 문서는 단순 필드 목록이 아니다.이 문서는 정보구조, UX, 카피, AEO, 운영 workflow가 연결된 실전용 CMS 기준 문서다.
2. CMS 설계의 기본 원칙
2.1 실무 안정성이 최우선이다
Webflow에서는 너무 많은 다중 참조와 과도한 중첩 구조를 만들면, 디자이너·개발자·운영자 모두 헷갈리기 쉽다.따라서 1차 구축에서는 “이론적으로 가장 이상적인 구조”보다 운영이 쉬운 구조를 우선한다.
2.2 정본 페이지와 해설 페이지를 분리한다
Answer Card는 정책의 정본 페이지다.Issue Briefing은 기사형 해설 페이지다.Fact Check는 공식 검증 페이지다.Profiles는 신뢰·권위 페이지다.
즉 CMS도 이 역할 차이를 그대로 반영해야 한다.
2.3 컬렉션은 적게, 필드는 명확하게
컬렉션을 불필요하게 많이 만들기보다, 핵심 컬렉션을 분명히 정의하고 필드 설명을 자세히 적는 편이 더 낫다.
2.4 공개 상태를 엄격하게 제어한다
현재 사이트에는 샘플 카드와 범용 브리핑이 공개되어 사이트 엔티티를 흐리고 있다.따라서 모든 주요 컬렉션에는 Public Status 또는 동등한 상태 관리 필드가 반드시 있어야 한다.
2.5 검수자는 김태영 교수 기준으로 통일한다
향후 Answer Card, 이슈 브리핑, 팩트체크, 인증센터 관련 모든 기본 검수자 필드는 김태영 교수 기준으로 설정한다.기존 이한주 표기는 기본값에서 제거한다.
3. 최종 CMS 구성 개요
hanjunho.ai의 1차 운영 기준 핵심 컬렉션은 다음 4개다.
Profiles
Answer Cards
Issue Briefings
Fact Check Items
이 4개가 핵심 동적 컬렉션이다.
반면 다음은 정적 페이지 중심으로 시작하는 것이 좋다.
Ask AI 홈5대 비전 클리어맵 인덱스 프레임팩트 인증센터 메인 페이지투표 안내질문·제안·제보 폼 페이지
즉, 동적 데이터는 4개 컬렉션이 담당하고,전역 허브 페이지는 정적 페이지 + 컬렉션 리스트 조합으로 운영하는 것이 가장 안정적이다.
4. 컬렉션 간 관계 구조
먼저 전체 관계를 문장으로 정리하면 다음과 같다.
Profiles 컬렉션은Answer Cards의 Author, Reviewer로 연결된다.Issue Briefings의 Author, Reviewer로 연결된다.Fact Check Items의 Author, Reviewer로 연결된다.
Answer Cards 컬렉션은Issue Briefings의 Related Cards와 연결된다.Fact Check Items의 Related Cards와 연결된다.Profiles의 Related Cards와 연결될 수 있다.
Issue Briefings 컬렉션은Answer Cards와 다대다 관계를 가진다.Profiles와 작성/검수 관계를 가진다.
Fact Check Items 컬렉션은Answer Cards 및 Issue Briefings와 연결될 수 있다.
핵심은 다음과 같다.
Profiles는 신뢰 주체 컬렉션이고,Answer Cards는 정책 정본 컬렉션이며,Issue Briefings는 기사형 해설 컬렉션이고,Fact Check Items는 검증 컬렉션이다.
5. 공통 필드 운영 규칙
각 컬렉션마다 일부 공통 필드를 둬야 운영이 편하다.
5.1 Public Status
모든 핵심 컬렉션에 공통으로 둔다.권장 값은 다음과 같다.
DraftReviewPublicHiddenArchived
Draft는 작성 중인 상태다.Review는 내부 검토 또는 김태영 교수 검수 전/중 상태다.Public은 공개 상태다.Hidden은 공개 중단 상태다.Archived는 보존은 하되 운영에서는 제외하는 상태다.
5.2 Featured on Home
홈에 노출할지 여부를 결정한다.Home 대표 카드, 최신 브리핑, 주요 팩트체크 노출 제어에 사용한다.
5.3 Display Priority
정렬 우선순위를 숫자로 지정한다.숫자가 낮을수록 먼저 노출되게 운영하는 것이 좋다.
5.4 Last Updated
모든 주요 상세페이지에 최신성을 보여주기 위해 필수다.
5.5 Meta Title Override / Meta Description Override / OG Image
페이지별 메타와 공유 이미지를 세밀하게 제어하고 싶을 때를 대비해 권장한다.초기에는 비워 두고 자동 규칙을 쓸 수 있지만, 고도화 단계에서는 꼭 필요해진다.
6. Profiles 컬렉션 사전
6.1 컬렉션 목적
Profiles는 hanjunho.ai 안에서 누가 답을 말하고, 누가 검수하고, 누가 책임지는가를 설명하는 핵심 컬렉션이다.
이 컬렉션은 다음 페이지에서 사용된다.
팩트 인증센터 메인한준호 프로필 상세김태영 교수 프로필 상세Answer Card 상세 하단 작성자·검수자 블록Issue Briefing 상세 하단 작성자·검수자 블록Fact Check 상세 하단 작성자·검수자 블록
6.2 운영 핵심 아이템
Profiles 컬렉션의 핵심 아이템은 다음 세 개다.
한준호김태영 교수한준호 캠프 AI 정책본부
필요하면 추후 추가할 수 있지만, 1차는 이 세 개만으로도 충분하다.

10. 정적 페이지와 CMS의 역할 분담
CMS가 모든 것을 처리할 필요는 없다.오히려 1차 구축에서는 다음처럼 역할을 나누는 것이 안정적이다.
홈은 정적 페이지로 두되, 대표 카드·대표 브리핑은 CMS 리스트로 불러온다.5대 비전 클리어맵은 정적 프레임 + Answer Cards 컬렉션 리스트로 구성한다.이슈브리핑 인덱스는 정적 프레임 + Issue Briefings 리스트로 구성한다.팩트 인증센터 메인은 정적 페이지로 두고 Profiles 리스트를 불러온다.투표 안내는 정적 페이지로 시작한다.질문·제안·제보는 정적 폼 페이지로 운영한다.
즉, CMS는 정본 콘텐츠를 관리하고, 상위 허브는 정적 페이지가 방향을 잡는 구조가 가장 좋다.

첫째, Profiles 컬렉션 정리한준호 / 김태영 교수 / 한준호 캠프 AI 정책본부세 아이템 고정
둘째, Answer Cards 컬렉션 정리Category 오분류 수정Policy Domain 채우기김태영 교수 Reviewer 적용샘플 카드 Hidden 처리
셋째, Issue Briefings 정리범용 브리핑 Hidden 처리대표 기사 정보 입력Related Cards 연결
넷째, Fact Check Items 구축핵심 쟁점 몇 개부터 시작
12. 최종 운영 주의사항
필드가 많아질수록 운영이 복잡해질 수 있다.따라서 운영팀은 무조건 모든 필드를 다 채우려고 하기보다, 필수 필드와 권장 필드를 구분해 입력해야 한다.
필수 필드가 비어 있으면 공개하지 않는다.권장 필드는 1차 런치 후 단계적으로 보완할 수 있다.
특히 다음 항목은 공개 전 반드시 확인해야 한다.
Answer Card의 Category와 Policy Domain이 맞는가Reviewer가 김태영 교수 기준으로 연결되었는가Issue Briefing에 기사 출처와 Related Cards가 있는가Profiles의 Role과 Summary가 상단 역할 정의 문장으로 충분한가Public Status가 정확한가
13. 최종 요약
이 CMS 사전의 핵심은 다음과 같다.
Profiles는 신뢰 주체 컬렉션이다.Answer Cards는 정책 정본 컬렉션이다.Issue Briefings는 기사형 해설 컬렉션이다.Fact Check Items는 공식 검증 컬렉션이다.
그리고 이 네 컬렉션이 한준호.ai 전체의 홈, 클리어맵, 상세페이지, 인증센터, 팩트체크, 프로필 페이지를 떠받치는 데이터 구조가 된다.
이 문서는 이후페이지별 구현 명세서,카피 마스터 문서,AEO·SEO·Schema 기술 명세서작성의 기준 문서로 사용한다.
